Memo

개요

Memo는 로그인이나 서버 의존성 없이
브라우저만으로 즉시 사용할 수 있는 휘발성 메모·플래너 애플리케이션이다.

이 프로젝트의 핵심 목적은 기능 구현 자체가 아니라,
**컴포넌트 주도 개발(Component Driven Development)**을 실제 제품 단위로 적용하고
UI 설계·상태 분리·확장 가능성을 검증하는 데 있다.

문제 정의

프론트엔드 개발 과정에서 다음과 같은 구조적 한계를 자주 경험했다.

  • UI와 비즈니스 로직이 강하게 결합된 컴포넌트 구조
  • 페이지 단위 개발로 인해 컴포넌트 재사용성이 낮아짐
  • 상태 케이스(빈 상태, 긴 텍스트 등)를 사후에 대응
  • 작은 서비스임에도 과도한 아키텍처 도입

Memo는
“작은 제품에서도 설계는 분리되어야 한다”는 가설을 검증하기 위한 실험이었다.

해결 전략

1. Component Driven Development (CDD)

Storybook을 중심으로 개발 프로세스를 설계했다.

  • 페이지 이전에 컴포넌트 단위 설계
  • 메모 아이템, 입력창, 버튼 등을 독립된 상태로 개발
  • Empty / Filled / Edge Case를 스토리로 사전 정의

이를 통해 UI는 비즈니스 흐름과 분리된 채로 검증되었고,
페이지 조립은 상대적으로 단순해졌다.

2. TypeScript 기반 데이터 모델링

메모 데이터 구조와 컴포넌트 Props를 명확히 정의했다.

  • 메모 아이템 인터페이스 통일
  • 상태 전달 과정에서의 휴먼 에러 차단
  • UI 변경 시 영향 범위를 컴파일 단계에서 인지 가능

작은 앱이지만, 확장 가능한 데이터 모델을 전제로 설계했다.

3. 휘발성 플래너 컨셉

Memo는 장기 보관용 노트가 아니라
빠르게 쓰고, 쉽게 버리는 메모에 초점을 맞췄다.

  • 복잡한 계층 구조 제거
  • 즉각적인 추가 / 삭제
  • 단기 일정과 생각 정리에 최적화된 UX

기능을 늘리기보다 “지우기 쉬운 도구”라는 성격을 명확히 했다.

4. PWA 기반 오프라인 경험

  • Service Worker를 통한 캐싱
  • 네트워크 없이도 메모 작성 및 조회 가능
  • 홈 화면 추가를 통한 앱 수준 접근성 제공

또한 Download 기능은 파일 저장이 아닌
PWA 설치 또는 모바일 앱 다운로드 링크로 연결되도록 설계했다.

기술적 선택 근거

  • TypeScript
    → 컴포넌트 Props와 상태 모델을 명확히 관리하기 위함

  • Storybook
    → UI를 코드 이전에 검증하고, 문서로 동시에 활용하기 위함

  • PWA
    → 서버 없는 환경에서도 앱에 가까운 사용자 경험 제공

  • Vanilla HTML / CSS
    → UI 프레임워크에 의존하지 않고 구조·스타일 설계 역량을 집중하기 위함

  • LocalStorage
    → 즉각적인 데이터 반응성과 단순한 상태 저장을 위해 채택

결과 및 의의

  • CDD 방식의 실제 제품 적용 경험 확보
  • 컴포넌트 단위 설계 → 페이지 조립 흐름 정립
  • 과하지 않은 아키텍처로도 유지보수 가능한 구조 검증
  • 이후 프로젝트(UI 시스템, 디자인 시스템)의 기반 사례로 활용 가능

Memo는
**“작은 서비스에서의 올바른 프론트엔드 설계”**를 검증한 작업이다.
